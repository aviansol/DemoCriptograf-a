Selected AEAD Algorithm: AES-GCM-256
AES in Galois/Counter Mode provides both confidentiality and integrity in a single pass. The 256-bit key offers 128 bits of security against exhaustive search.

Key Size: 256 bits (32 bytes)
Generated via AESGCM.generate_key(bit_length=256) which uses OS-provided CSPRNG (os.urandom internally).

Nonce Strategy: 96-bit random nonce per encryption
A fresh 12-byte nonce is generated with os.urandom(12) for every encryption call. The nonce is stored alongside the ciphertext in the container.

Why AEAD instead of encryption + hash?
A naive Encrypt-then-MAC construction requires two separate keys and careful ordering. AES-GCM provides an atomic authenticated encryption: the same operation that produces ciphertext also produces an authentication tag covering both the ciphertext and any associated data (metadata). There is no gap between encryption and authentication where an attacker can insert data.

What happens if the nonce repeats?
Nonce reuse with the same key in GCM is catastrophic: an attacker can XOR two ciphertexts to eliminate the keystream, recovering the XOR of both plaintexts. Worse, they can forge authentication tags, breaking integrity entirely. This is why each encryption uses a fresh random nonce.

Metadata Authentication (AAD):
Filename, algorithm version, key size, nonce size, and creation timestamp are serialised as JSON and passed as Associated Authenticated Data (AAD). AES-GCM's authentication tag covers this data, so any silent modification to the metadata causes decryption to fail.

Attacker Model:
We defend against a ciphertext-only and chosen-ciphertext attacker who has obtained the encrypted container (file at rest) but does not have the key. They may attempt to read, modify, or replay the ciphertext or its metadata. AES-GCM ensures both goals fail: without the key, they cannot decrypt; any modification is detected by the authentication tag.

Container structure: header (metadata + AAD) | nonce (96-bit) | ciphertext | auth_tag (128-bit)
